JS is the problem, not HTML or CSS. The 5 biggest functions alone account for ~120 KB.

Also note: the browser never downloads 889 KB. GitHub Pages serves with gzip — JS compresses ~75%, so the wire size is probably ~200–220 KB. The authoring problem (hard to navigate, edit, reason about) is different from the load problem.

Two distinct problems — pick which one you want to solve:

Problem 1: Authoring (hard to work in)
Build-time splitting — source lives in many small files, a build step (simple build.sh) concatenates them into index.html. Zero change to runtime behaviour, state, or architecture. S stays global. Nothing breaks.

Problem 2: Load performance (slow initial load)
Not actually a problem right now because of gzip + service worker caching. After the first load, the SW serves from cache. If it becomes a problem, runtime lazy loading via ES modules: navTo() triggers import('./sections/bottles.js') on first navigation. Bigger architectural change.

Recommendation: Start with Problem 1 only — build-time splitting. The Eleventy file in the repo is for a separate blog idea, not relevant here. Right tool: simple build.sh, not Eleventy or webpack.
