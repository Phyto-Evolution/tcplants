Here's what actually happens, in order, for each mode:

## Single bottle

You fill the form — species, stage, status, recipe, accession, volume, explants, dates, next review, notes, flair. Hit save.

`saveBottleAction()` reads every field from the DOM. Assembles one `base` object. Since `qty` isn't read in single mode, it hardcodes `qty=1`. Generates one code: takes every existing bottle's code, parses the number out of `FSL-YYYY-NNNN`, finds the max, adds 1. So the new code is always `FSL-<current year>-<max+1>`. No `batch_id` since qty is 1.

Pushes that one object into `S.bottles`, calls `saveBottles()`. That function packages the entire bottles array plus all sessions into one JSON object, AES-256-GCM encrypts the whole thing, base64-encodes it, and does a GitHub `PUT` to `bottles/data.enc` with the SHA from the last load. If something else wrote to that file since you loaded, GitHub returns 422 and you get "Save conflict" — you'd have to try again. On success it gets back the new SHA and stores it in `S.bottlesSha`. Then `renderBottleList()` rerenders the list and `openBottle()` jumps straight to the new bottle's detail view.

## Batch

Same as single but you've set mode to "batch" and entered a qty. `S._botMode === 'batch'` so the qty field is read. Say you type 20. The code loops 20 times, each iteration calling `_botNextSeq() + i` — so seq starts at max+1 and each new bottle increments it. All 20 get the same `batch_id` (one `uid()` shared), the same `base` fields (species, stage, recipe — everything identical). They all go into `newBottles`, appended to `S.bottles`, saved in one single `ghPut` call with the whole array encrypted together.

After save it opens `newBottles[0]` — the first bottle of the batch — in the detail panel.

**What you do not get:** no per-bottle variation. All 20 are identical except their codes and ids. No way to give different explant counts, different recipes, or different species to individual bottles within a batch from this form. That's what session mode is for.

## Session (autoclave session)

Completely different path. The regular form fields are hidden by CSS (`bot-mode-session` class suppresses `.field`, `.row2`, `.row3`). Instead you get the session builder: a date, autoclave capacity, temp, time, and a list of batch rows.

Each batch row has: species, stage, recipe, and a qty. You can add as many rows as you want with "+ Add batch". The capacity bar shows total vs. autoclave capacity live, turns yellow at 90%, red when over.

Hit save → `saveSessionAction()`. It reads all batch rows from the DOM by querying `.sess-batch-row` elements. For each batch row it creates a `batch_id` (per row, not per session). Then it loops `qty` times per batch, generating a bottle for each with `FSL-YYYY-NNNN` codes where the sequence counter increments continuously across all batches in the session. All bottles in a batch share a `batch_id`; all bottles in the session share a `session_id`.

The session itself (date, capacity, autoclave params, the batch definitions) gets pushed into `S.sessions` as a separate record. `saveBottles()` is called with both the updated bottles array and the updated sessions array — both go into the same `bottles/data.enc` file encrypted together.

**What a session bottle gets that a single/batch bottle doesn't:** `session_id` field, `batch_id` field. But it doesn't get `next_review`, `explants`, `media_volume`, `accession_id`, `note_id`, `flair_ids`, `greenhouse_plant_id` — those fields don't exist in the session builder form. The session path hardcodes `status: 'prepared'`, `media_volume: 25`, and nothing else. Those fields are only in the regular single/batch form.

## The shared problem across all three

Every save — single, batch, session — reads the full `S.bottles` array, re-encrypts the entire thing, and writes it back to GitHub as one atomic PUT. There is no partial update. So if you have 500 bottles and add 1, the entire 500+1 get re-encrypted and PUT. The SHA must match what GitHub has or it rejects it. There is no merge, no retry logic, no read-before-write on conflict — if you get a 422 the error just surfaces as a toast and nothing is saved.
